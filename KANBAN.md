# nio-flow ‚Äî Kanban

Feature board for nio-flow: a library for complex business logic that stays easy to extend.
Every item is tagged by focus: `[perf]` performance, `[scale]` scalability, `[maint]` maintainability/DX.

Workflow rule (from CLAUDE.md): a feature is checked off only with unit tests in `core/` AND JMH
benchmarks in `tests/` showing good results ‚Äî no hot-path regressions.

- [x] **Event loop engine** ‚Äî boss pool with per-execution affinity + virtual-thread workers; boss never runs user code `[perf] [scale]`
- [x] **JVM-shared executors** ‚Äî commonPool-style daemon bosses/workers; ownership-aware `shutdown()` `[scale]`
- [x] **Stage + filter fusion** ‚Äî one boss‚Üíworker‚Üíboss round-trip per run of cheap links (~5.7x on 8-stage chains) `[perf]`
- [x] **Runtime chain editing** ‚Äî `splice` BEFORE/AFTER/REPLACE on named stages; immutable chain + per-call snapshot: in-flight requests never affected `[scale] [maint]`
- [x] **Per-request executions** ‚Äî `just()...execute()` on a shared singleton bean, fully isolated between concurrent requests `[scale]`
- [x] **Forks `when`/`match`** ‚Äî first-match-wins cases, nested forks, guards composed automatically; restricted `Lane<T>` (no execute/just/close inside a branch) `[maint]`
- [x] **`filter`** ‚Äî short-circuit on shared or per-request chains, fused into worker runs `[perf]`
- [x] **`background`** ‚Äî fire-and-forget side effects; errors reported to handlers, never fail the flow `[scale]`
- [x] **Recovery links (engine)** ‚Äî positional error handling, catches stage failures and timeouts `[maint]`
- [x] **`recover` in the fluent API** ‚Äî `recover(fn)` / `recover(name, fn)` on `NioFlow` and `Lane` (lane-scoped via guards); recoveries fuse into worker runs: happy path at parity with plain stages, error path 2x faster `[maint] [perf]`
- [x] **`executeAsync()` returning `CompletableFuture<T>`** ‚Äî non-blocking endpoints by returning the future from the controller; `execute()` is now `executeAsync().join()`; single-call parity, 2.6x on 16 pipelined executions `[scale]`
- [x] **Stage timeout in the fluent API** ‚Äî `handle(name, fn, Duration)` on `NioFlow` and `Lane`; timeout failures are recoverable downstream; armed budget costs ~40% on that stage only (fusion break + orTimeout timer, by design) `[maint]`
- [x] **Backpressure for `inject`/`justAll`** ‚Äî `DefaultNioEngine(capacity, OverflowPolicy)`: BLOCK parks the producer, DROP discards (reported to error handlers), FAIL throws; admission happens BEFORE the execution runs, slots free on `await()`; uncontended cost at parity with unbounded `[scale]`
- [x] **Metrics SPI + OpenTelemetry adapter** ‚Äî `NioFlowMetrics` (no-op defaults) installed via `engine.metrics()`: execution/stage latency, failed/filtered classification, recoveries, drops, queue depth; `infrastructure/OpenTelemetryMetrics` exports histograms/counters/gauge with cached attributes; instrumentation cost at parity with metrics off `[maint] [scale]`
- [x] **Chain compilation at `seal()`** ‚Äî precomputed fusion windows and unguarded runs per chain version; splice recompiles once per edit; execution-local chains fall back to interpreting. Measured honestly: throughput at parity (thread hops dominate ‚Äî the "biggest win" framing was wrong), but **~13% less garbage per request** (1170 ‚Üí 1017 B/op) and the compiler pass is the foundation for seal-time validation and the decisions bitset `[perf]`
- [x] **`fanOut`/`fanIn`** ‚Äî `fanOut(name, branches, join)` on `NioFlow` and `Lane`: branches run concurrently on workers (one each), join combines in declaration order on a worker, failures recoverable downstream; 2.2x on 3√ó50¬µs branches, ~28% overhead on trivial branches (use it for real work); works in lanes and on compiled chains `[scale]`
- [x] **Reusable sub-flows (`Segment<T, R>` + `use`)** ‚Äî a segment defines a chain piece over `Lane<T>` ending at `Lane<R>`; embedded inline with the caller's guards (lane-scoped when used in forks), segments compose and are independently testable; build-time only, runtime at parity with inline definitions `[maint]`
- [x] **Graceful drain on shutdown** ‚Äî `shutdown(grace)` now returns the pending count: rejects new call/inject immediately, waits for in-flight executions up to the grace (works for JVM-shared executors too ‚Äî the old code was a no-op there), then terminates engine-owned executors; stragglers on shared executors still finish on their own; hot-path counter at parity `[scale]`
- [x] **Distinguish filtered from null results** ‚Äî public `FlowSignal.FILTERED` carried by raw engine futures (engine exits map it to null); `executeResult()` returns sealed `FlowResult<T>` (`Completed(value)` ‚Äî even genuinely null ‚Äî vs `Filtered`), pattern-matchable; `execute`/`executeAsync` keep the null mapping for compatibility; filter paths at parity `[maint]`
- [x] **Retry policy per stage** ‚Äî native `Retry` (attempts + backoff + multiplier, zero external deps: Resilience4j stays an optional adapter for circuit breaker/bulkhead); `handle(name, fn, Retry)` / `handle(name, fn, Duration, Retry)` on `NioFlow` and `Lane`; no-timeout retries loop inline on the (virtual) worker and never break fusion, timeout+retry applies the budget per attempt with non-blocking backoff scheduling; exhausted retries flow to recovery; observable via `stageRetried` metrics; declared-but-unused at parity, one retry costs ~15% `[scale]`
- [x] **Validation at `seal()`** ‚Äî `ChainValidationException` (with the full problem list) on dangling guards, contradictory guards, duplicate anchor names and dead recoveries; runs at seal (broken definitions stop the deploy) and on every splice over a sealed chain (a rejected runtime edit leaves the previous chain and plan untouched); build-time only, runtime at parity `[maint]`
- [x] **Decisions as bitset** ‚Äî per-execution decisions in a `long[]` bitset (2 bits per id: recorded + value) instead of `HashMap<Integer, Boolean>`: O(1) records/guards with zero per-decision allocation, and an unrecorded id still fails any guard (what first-match-wins relies on). Sized by the chain's highest Decision id (precomputed at `seal()`, scanned once per call on interpreted chains); ids past 511 ‚Äî per-request forks on a long-lived engine keep growing the engine-wide counter ‚Äî fall back to a lazy overflow map with identical semantics. Throughput at parity; ~14% less garbage per forked request (1192 ‚Üí 1025 B/op on match routing, 1312 ‚Üí 1144 B/op on forked compiled chains) and decision-free executions drop the eagerly-allocated HashMap (~48 B/op) `[perf]`
- [x] **Inline cheap stages on the boss (opt-in)** ‚Äî `handleSync(name, fn)` on `NioFlow` and `Lane` runs a pure-CPU, sub-microsecond stage inline on the boss, skipping both thread hops (same contract as Decision predicates: never block; a throw fails the value through recovery, never the boss task). 2.2-2.4x on trivial chains (1 stage: 56 ‚Üí 126 ops/ms; 4 stages: 57 ‚Üí 135 ops/ms) with ~59% less garbage (1001 ‚Üí 416 B/op); a sync stage after a worker stage still fuses into that run. No timeout/retry variants by design ‚Äî hand-built sync+timeout/retry stages are rejected at seal(). Worker-stage paths at parity (the extra branch measured nothing) `[perf]`
- [x] **Boss pool tuning** ‚Äî `DefaultNioEngine.dedicated(bossCount)` (and a backpressure variant) gives an engine its OWN boss pool + virtual worker pool: no other engine queues orchestration behind its bosses, and `shutdown()` terminates them (shared engines unaffected); executions keep one-boss affinity. The JVM-shared pool size is tunable via `-Dnioflow.bosses=N` (default: cores, floor 2). Measured where it matters ‚Äî tail latency of a victim flow beside a 7-thread noisy neighbor: p99 162 ‚Üí 114¬µs (‚àí30%), p99.9 293 ‚Üí 152¬µs (‚àí48%), p99.99 1.29 ‚Üí 0.51ms (‚àí61%); uncontended single-call at parity with shared (57.6 vs 55.2 ops/ms ‚Äî same dispatch code, only the pool changes) `[scale]`
- [x] **Context API** ‚Äî `Context` with typed name-based keys (`Context.Key<T>`, `ctx.get(key)`/`put`/`getOrDefault`); stages opt in via `handleContextual(name, (value, ctx) -> ...)` on `NioFlow` and `Lane` (a separate method, not a `handle` overload ‚Äî inexact method refs like `String::toUpperCase` would turn ambiguous and break existing code). Keys are name-based so a map handed to `engine.call(input, map)` interoperates. Build-time the BiFunction wraps in an internal `ContextualFunction`; the engine unwraps it at the single stage-apply point (one `instanceof` for plain stages) and the backing HashMap lazy-inits on the first put ‚Äî stage applications are serialized by the executor handoffs, no synchronization needed. Plain and sync paths at exact parity (732 / 256 B/op, re-measured against same-day baseline); a contextual chain costs +159 B/op with throughput parity `[maint]`
- [x] **Flow-level `onComplete`/`onError` in the fluent API** ‚Äî on the shared definition they tap the engine handlers (every execution reports; `onError` is the engine's error tap ‚Äî rejected/dropped values and failing background effects included); on a `just()` execution they scope to that execution and fire BEFORE `execute()` returns; multiple callbacks compose in declaration order. Hardened alongside: handlers can no longer break completion ‚Äî a throwing complete handler is reported through the error handlers, a throwing error handler is contained (previously it could leave the caller's future hanging). Costs: no callbacks 730 B/op (parity with the post-audit 727 baseline), shared callback zero extra allocation per request, execution-scoped callback +112 B/op (its dependent future) `[maint]`
- [x] **Allocation audit per call** ‚Äî three eager allocations removed from the common path: the per-call `ConcurrentHashMap` context nobody reads yet (stays null until the future Context API lazy-inits it), the dependent `whenComplete` future per call (bookkeeping ‚Äî drain slot, metrics, handlers ‚Äî moved inside `Execution`, running BEFORE the raw result future completes, same caller-observable order; `Execution` itself is now the initial boss `Runnable`), and the `CompletableFuture` machinery per dispatch (the boss‚Üíworker‚Üíboss hop is a manual two-closure handoff; CF stays where its semantics pay: timeout `orTimeout`, fan-out `allOf`). A boss executor shut down mid-flight now fails the future instead of hanging it. ‚àí23-40% garbage across the board: plain 1001 ‚Üí 727 B/op, forked 1144 ‚Üí 866, match routing 968 ‚Üí 750, sync 416 ‚Üí 248, contended 979 ‚Üí 700; throughput at parity or better. The remaining ~250 B/op floor is the API surface itself (result future, Execution, per-request flow view) `[perf]`
- [x] **Fusion across recorded decisions** ‚Äî compiled fusion windows now extend through *guarded* non-fusable links (a `match()` case's Decision the execution's recorded decisions already ruled out, skipped lanes): the per-execution scan steps over guard-failed links, a *passing* Decision still ends the run, and an unguarded one stays a hard boundary ‚Äî converging with what the interpreted scan already did. A case-1 route over `match(3) + 2 tail stages` (sealed) drops from 2 dispatches to 1: 35.0 ‚Üí 54.6 ops/ms (+56%), 1612 ‚Üí 1152 B/op (‚àí29%), reaching parity with the already-fused otherwise route; plain chains and already-fused routes at parity `[perf]`
- [x] **Boss safety invariants** ‚Äî iterative `advance` (no stack overflow on deep chains), throwing predicates fail the value never the boss task `[scale]`
- [x] **Quality harness** ‚Äî JMH benchmarks (`tests/`), bug-hunting stress tests, Spring Boot showcase example `[maint]`
