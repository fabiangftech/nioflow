## Feature workflow

Every new feature must ship with unit tests (in `core/`) AND JMH benchmarks (in `tests/`) showing good results: run the relevant benchmarks before and after, compare, and report the numbers. A feature that regresses the hot path is not done тАФ either fix it or surface the trade-off explicitly. Bug-hunting stress tests in `tests/` complement, not replace, functional coverage in core.

**Never skip the static analysis.** Before committing any code change, run SonarLint over every module you touched and fix what your change introduced:

```bash
tools/sonarlint/run.sh          # core
tools/sonarlint/run.sh tests    # the benchmarks/stress module
```

Judge by the DIFF, not the total: the repo carries known findings it violates on purpose (`tools/sonarlint/README.md` lists them тАФ do not "fix" those). To see only what you added, run the same script over a worktree of the previous commit (`git worktree add /tmp/base HEAD~1 && tools/sonarlint/run.sh /tmp/base/core`) and diff `tools/sonarlint/build/issues.json`. New issues in your files get fixed before the commit; if one is a deliberate design decision, it goes in that README with the reason.

## Project

nio-flow is a Java concurrency library: a fluent, typed pipeline API (`NioFlow<I, O>` for the shared definition, `NioStep<T, O>` for the per-request pipeline) over an event-loop engine (a pool of boss threads orchestrates тАФ each execution pinned to one boss тАФ while virtual-thread workers run user code). Zero required runtime dependencies (Resilience4j and OpenTelemetry are `compileOnly` integrations). Requires a modern JDK тАФ uses virtual threads and pattern matching over sealed types; developed on JDK 25.

## Architecture

Three packages under `core/src/main/java/dev/nioflow/`, dependency direction strictly inward (`application` and `infrastructure` depend on `core`, never the reverse). Interfaces live in `core.facade`; implementations in `application.facade` with a `Default` prefix.

- **`core/facade`** тАФ public contracts, split by role. `NioFlow<I, O>` is the **shared definition** (the bean): `I` is what `just()` accepts, `O` is what the per-request pipeline must return. Its building steps are **type-preserving over `I`** (there is no `adapt` on it, and `fanOut`/`batch`/`use` give the value back as `I`) тАФ that invariant is what lets `just()` hand back a pipeline that starts at the input type. `NioStep<T, O>` is that **per-request pipeline**: `T` is the value's type right here, `adapt`/`fanOut`/`batch`/`use` move it, and the terminals (`execute`/`executeAsync`/`executeResult`) return `T` тАФ so a pipeline that has not reached `O` fails to compile at the caller's return statement. `O` is a documented contract (Java cannot demand fluently that a chain end at `O`); it is only read by `NioFlow.onComplete(Consumer<O>)`. The split also turns old runtime traps into compile errors: `execute`/`key` do not exist on the definition, `justAll`/`use(region, тАж)` do not exist on a pipeline. Lifecycle lives only on `DefaultNioFlow.close()` (the engine owner). `NioEngine` is the engine contract behind it тАФ untyped (`Object`) on purpose; all unchecked casts are encapsulated in the flow implementations. `Condition`/`Branch`/`Cases` are the branching contracts on the definition (lanes over `Lane<I>`); `StepCondition`/`StepBranch`/`StepCases` their per-request counterparts (lanes over `Lane<T>`); `Lane`/`LaneCondition`/`LaneBranch`/`LaneCases` are the restricted lane-side ones, shared by both. **The type-preserving steps take `UnaryOperator`, not `Function`**: `handle`/`handleSync` (and `NioFlow.batch`, whose bulk stays at `I`) тАФ the signature says what the step must not do (re-type the value), and `adapt`/`fanOut`/`recover`, which genuinely change the type, keep taking `Function`. A caller composing with `andThen`/`Function.identity()` gets a `Function` back and has to adapt it (`f::apply`); that is the intended friction. The `Batch` link's `bulk` field stays a `Function` on purpose (see `tools/sonarlint/README.md`).
- **`core/model`** тАФ the chain model: `Link` is a sealed interface permitting `Stage` (transform, optional timeout), `Decision` (records a boolean per value), `Filter` (short-circuits the flow), `Background` (fire-and-forget side effect), `Recovery` (positional error handler), `FanOut` (parallel split-join: branches run concurrently on workers, join combines in declaration order; a branch failure fails the fan-out, recoverable downstream), `Batch` (coalescing point: executions park until size/window, one positional bulk call, each caller gets its own result; a CLASS with identity equality on purpose тАФ the engine keys in-flight groups by link instance), `Fork` (detached sub-flow: carries an immutable, guard-closed sub-chain the engine runs as an independent child execution тАФ see below). Every link carries `Guard`s (`decision id` + `expected`) for lane routing. `Splice` (BEFORE/AFTER/REPLACE) names the runtime-edit positions. FanOut breaks fusion runs (it is a dispatch boundary) and, due to Java inference limits, callers should type the branches list explicitly (`List<Function<T, R>> branches = List.of(...)`) so the join lambda infers `List<R>`.
- **`application/facade`** тАФ `DefaultNioEngine`; `AbstractChain<X>` (the untyped chain-building core: every link and every unchecked cast lives there) with three typed facades on top тАФ `DefaultNioFlow<I, O>` (definition), `ExecutionNioFlow<T, O>` (per-request pipeline) and `DefaultLane<T>` (branch lanes and fork sub-flows); `RecordingChain` (off-chain recorder for `replaceRegion` and for a fork's sub-chain); branching impls `Default{,Step,Lane}{Condition,Branch,Cases}`, with `NioFlowDelegate`/`NioStepDelegate` carrying the "chaining after a branch returns to the main line" plumbing.
- **`infrastructure`** тАФ optional adapters over `compileOnly` dependencies: `OpenTelemetryMetrics` (metrics SPI тЖТ otel histograms/counters/gauge; only loads if the consumer brings `opentelemetry-api`), `Resilience4jStages` (circuit breaker / bulkhead decorators for stage functions; breaker-open and bulkhead-full failures flow to `recover()` like any stage failure, each `Retry` attempt passes the decorators, bulkhead waits park the virtual worker; only loads if the consumer brings resilience4j) and **`infrastructure.reactive`** (WebFlux тАФ see below; only loads if the consumer brings `reactor-core`).

### DefaultNioEngine: the event loop

Two rules define it:

1. **Each execution is pinned to one boss, and only that boss touches its orchestration state.** `Execution.advance`/`recover` always run on the execution's boss; workers hand results back via `whenCompleteAsync(..., boss)`. This is the serialization mechanism тАФ no locks in the hot path. Concurrent executions spread across the boss pool (EventLoopGroup-style affinity), so one boss is never a JVM-wide ceiling.
2. **The boss never runs user code** тАФ `Stage`/`Background`/`Recovery` functions go to the virtual-thread workers. Exceptions by design: `Decision` predicates (and `Filter` predicates not absorbed into a fused run) run on the boss, and `handleSync(name, fn)` opts a pure-CPU sub-microsecond stage into boss-inlining (skips both thread hops, ~2.2x on trivial chains; still fuses into a preceding worker run; no timeout/retry тАФ validation rejects that combination). Both must stay cheap and non-blocking (same rule as Netty handlers). User code that throws on the boss fails the value through the recovery path тАФ never the boss task (a dead boss task means a forever-hanging request future).

Hot-path rules the benchmarks enforce (see `tests/`):

- `advance` must stay **iterative**, never recursive: cheap links (Decision/Filter/Background/guard-skips) are walked in a loop on the boss, and a recursive version overflows the boss stack on deep chains, killing the task and leaving the request future hanging forever (regression: `DeepChainStressTest`).
- **Stage fusion**: a run of consecutive no-timeout `Stage`s, `Filter`s and `Recovery`s travels bossтЖТworkerтЖТboss as ONE composed function (2 thread hops per run, not per link тАФ measured ~5x on an 8-stage chain, ~1.5x on stage-filter-stage). Fused `Filter` predicates evaluate on the worker; a rejection returns the internal `FILTERED` sentinel and completes the flow with `null`. Fused `Recovery`s keep positional semantics inside the run: a failure scans forward for the next in-run `Recovery` and continues from it; with none left it escapes the run and the boss scans the rest of the chain from the run's end тАФ equivalent, because the segment in between was already searched. Guard-skipped links inside the run are stepped over (decisions can't change until the next passing `Decision`, which ends the run). Stages with a timeout dispatch alone.
- No streams or allocations on the per-link path (`passesGuards` is a plain loop тАФ the stream version cost ~20% on branch routing).
- Per-call allocation discipline: the context map stays null until a contextual stage's first put (or a caller-handed map), `call()` returns the raw result future тАФ bookkeeping (drain slot, metrics, handlers) runs inside `Execution` BEFORE completion, so no dependent `whenComplete` future тАФ and the bossтЖТworkerтЖТboss dispatch is a manual two-closure handoff, not `CompletableFuture` composition (CF stays where its semantics pay: arbitrating the per-attempt timeout race, fan-out `allOf`). A boss rejection mid-flight fails the future, never hangs it. ~727 B/op on a plain sealed chain, 248 B/op boss-inlined.

Executors are JVM-wide singletons (`SharedExecutors` lazy holder, `commonPool()` style): a pool of daemon bosses (sized by available processors, floor 2 тАФ tunable with `-Dnioflow.bosses=N`) + one virtual-thread worker pool shared by every engine created with the default constructor, no matter how many `DefaultNioFlow`s exist. For latency-critical flows, `DefaultNioEngine.dedicated(bossCount)` (optionally with capacity+`OverflowPolicy`) gives that engine its own boss pool and worker pool: no other engine queues orchestration behind its bosses (measured: p99.9 тИТ48% beside a noisy neighbor), and `shutdown()` terminates them without touching the shared ones. `shutdown(grace)` is a **graceful drain**: it rejects new `call`/`inject` immediately, waits up to the grace for in-flight executions, and returns how many were still running (0 = clean). It only terminates executors that were explicitly passed in (`ownsExecutors`); shared ones survive тАФ closing one flow never starves the others, and stragglers finish on their own threads.

### Concurrency and runtime-editing invariants (the tests enforce these)

- The chain is an **immutable list swapped atomically** (`volatile`); `append`/`splice` build a new list under `synchronized`. Every `call()` snapshots the chain at submission, so a runtime `splice` never affects in-flight executions тАФ the next call sees the new chain.
- `seal()` **validates and compiles** the chain. Validation (`ChainValidator` тЖТ `ChainValidationException` with the full problem list) rejects dangling guards, contradictory guards, duplicate anchor names and dead recoveries тАФ a broken definition stops the deploy; a splice over a sealed chain validates too and, when rejected, leaves the previous chain and plan untouched. Compilation produces the dispatch plan (`CompiledChain`): static fusion windows plus precollected unguarded runs, so those dispatches do zero scanning/allocation (~13% less garbage per request; throughput parity тАФ hops dominate). Windows extend through *guarded* non-fusable links (e.g. a `match()` case's Decision the execution's recorded decisions ruled out) тАФ the per-execution scan steps over guard-failed links, a passing Decision still ends the run, and an unguarded non-fusable link is a hard boundary. `splice` recompiles once per edit; `append` invalidates; executions match the plan by chain identity, so per-request local chains fall back to interpreting. The plan is an optimization, never a semantic: compiled and interpreted must produce identical results (`DefaultNioEngineCompiledChainTest`).
- Each `call()` gets its own `Execution` (chain snapshot, decision bitset, result future): concurrent requests share nothing. Decisions are recorded in a `long[]` bitset тАФ 2 bits per id (recorded + value), sized by the chain's highest `Decision` id (precomputed in `CompiledChain`, scanned once per call on interpreted chains); an unrecorded id fails any guard on it (the property `match()` first-match-wins relies on), and ids past the bitset limit (per-request `when`/`match` keep growing the engine-wide counter) fall back to a lazy overflow map (a `Fork`'s sub-chain sidesteps this: its ids are private, so they are compacted to `0..n-1` at build time). Decision state is only ever touched on the boss.
- `seal()` blocks `append` (frozen definition) but **not** `splice` тАФ splice *is* the runtime-edit operation, anchored on `Stage`/`Background` names. `release()` re-opens appending. Multi-link edits go through REGIONS: `use(name, segment)` remembers the embedded span (boundaries by link identity) and `spliceRegion(name, links)` / `DefaultNioFlow.replaceRegion(name, segment)` swaps it atomically, re-pointing the region at the new links; `replaceRegion` records the segment off-chain via `RecordingNioFlow`, which draws decision ids from the live engine and shares the anonymous-name counter to avoid guard/anchor collisions.
- `Recovery` is positional: it catches failures (including `Stage` timeouts) from links upstream of it; execution continues after it with the recovered value. With no matching recovery, the failure reaches `errorHandlers` and the call's future. Declared fluently via `recover(fn)` / `recover(name, fn)` on `NioFlow` and `Lane` тАФ a lane-scoped recover inherits the lane's guards and only catches failures of values routed through that branch. Named recoveries are splice anchors like stages.
- Resilience composes in layers: **`RateLimit` gates admission тЖТ timeout per attempt тЖТ `Retry` over attempts тЖТ `recover()` as the final net**. `RateLimit` (`handle(name, fn, RateLimit)`) is a native token bucket refilled lazily from timestamps (no timer thread): `acquire()` parks the virtual worker until a token is due тАФ the boss never waits, fusion is preserved, and the wait is visible in the stage latency metric. One instance = one bucket; share it across stages to protect a single downstream. `Retry` is native (attempts + backoff + multiplier; Resilience4j remains an optional adapter concern): no-timeout retries loop inline on the virtual worker (`LockSupport.parkNanos` backoff) and never break fusion; timeout+retry arms the per-attempt budget on the shared `TimerWheel` (512 buckets ├Ч 10ms tick, one lazy daemon, MPSC staging: O(1) schedule/cancel instead of orTimeout's lock-guarded global heap; timeouts fire up to ~2 ticks late, never early) with non-blocking backoff scheduling тАФ backoff itself stays on `delayedExecutor` (cold path; a sub-tick backoff would degrade to wheel granularity). Retries are observable via `NioFlowMetrics.stageRetried`.
- `Filter` short-circuits by completing the raw engine future with the public `FlowSignal.FILTERED` sentinel; engine exits (`await`, complete handlers) and flow-level `execute()`/`executeAsync()` map it to `null`, while `executeResult()` returns a sealed `FlowResult<T>` (`Completed(value)` тАФ even genuinely null тАФ vs `Filtered`) for callers that need the distinction. `Background` never waits and never fails the flow; a throwing effect reports to `errorHandlers` only.
- Context API: `handleContextual(name, (value, ctx) -> ...)` on `NioFlow` and `Lane` gives a stage the typed per-execution `Context` (`Context.Key<T>` accessors, name-based so a map handed to `engine.call(input, map)` interoperates). Deliberately NOT a `handle` overload: inexact method refs (`String::toUpperCase`) would turn ambiguous between `Function` and `BiFunction` and break existing code. Internally the BiFunction wraps in `ContextualFunction` and the engine unwraps at the single apply point тАФ plain stages pay one `instanceof`, the map lazy-inits on first put, and executor handoffs serialize access (plain `HashMap`, no locks).
- `batch(name, size, window, bulk)` coalesces executions: they park at the link (still counted by the drain тАФ shutdown grace must cover the window), the window rides on the shared `TimerWheel`, the bulk runs on a worker and each execution resumes on its own boss with its own element. Bulk failures/wrong-sized results fail every batched execution, recoverable per execution. The group lock is the documented exception to the no-locks rule (guards only list adds/swaps). Belongs on the shared definition тАФ only executions of the same flow pool together.
- Keyed execution: `just(x).key(k)` pins same-key executions to a deterministic boss and runs them one at a time in submission order (per-key FIFO lane; the boss affinity is what makes the lane state lock-free). Distinct keys/unkeyed traffic keep full parallelism; lanes are removed when they drain; a failed execution releases its lane to the next. Head-of-line per key is the point тАФ stage timeouts bound it. `key()` only exists on executions (the shared definition throws).
- `inject`/`await` are the fire-and-forget pair (results queue up in `inFlight`); `call` is the request/response form returning a `CompletableFuture`. At the flow level, `execute()` blocks and `executeAsync()` returns the future (`execute()` IS `executeAsync().join()`) тАФ return the future from a controller for a non-blocking endpoint.
- `inFlight` is unbounded by default; `new DefaultNioEngine(capacity, OverflowPolicy)` bounds it with BLOCK (park the producer), DROP (discard, reported to error handlers) or FAIL (throw `RejectedExecutionException`). Admission happens BEFORE the execution starts тАФ rejecting an already-run value is not backpressure тАФ and the slot frees when `await()` collects the result.
- Observability: `engine.metrics(NioFlowMetrics)` installs the metrics SPI (no-op defaults, null = zero instrumentation). The engine pushes execution latency classified as completed/failed/filtered, per-stage latency (timed on the worker inside fused runs), applied recoveries, dropped values and queue depth. Fluent taps: `onComplete`/`onError` on the shared definition register engine handlers (every execution reports; `onError` also sees rejections, drops and background failures); on a `just()` execution they scope to that execution and fire before `execute()` returns. Callbacks run on engine threads: keep them fast and never throw тАФ if one throws anyway it is contained (a failing complete handler reports through the error handlers; handlers can never leave a result future hanging).

### DefaultNioFlow: shared definition vs per-request execution

The root `DefaultNioFlow` is the **shared definition** (a Spring singleton bean): `handle`/`background`/`adapt`/`filter` on it append to the engine's shared chain. `just(input)` opens an **independent ephemeral execution** (`ExecutionNioFlow`): it lazily copies the shared chain only if local links are added, and `execute()` runs it via `engine.call(input, context, chain)` тАФ never sealing or mutating anything shared. That is why N concurrent requests can each do `just(...)...execute()` on the same bean.

**The type parameters are enforced, not decorative.** `DefaultNioFlow.from(Class<I>)` is the recommended way to open a root flow (`create()` when the type cannot be named, e.g. a `NioFlow<?, ?>` bean); the constructors are private тАФ they used to let a caller claim a `DefaultNioFlow<Integer, String>` for an empty flow, a type that lies about what the pipeline returns. The `Class` token is kept, and `just`/`justAll` reject an input that is not an `I` with a clear `IllegalArgumentException`: the net for unchecked casts arriving from raw types or from frameworks that inject by generics (Spring happily resolves a `NioFlow<?, ?>` bean into ANY `NioFlow<X, Y>` field). The typed entry point is the factory:

```java
@Bean(destroyMethod = "close")
public NioFlow<Integer, String> credits() {
    return DefaultNioFlow.from(Integer.class);   // Integer in, String out
}
// per request: the pipeline starts at the INPUT type, and adapt takes it to the output
credits.just(cents)
       .handle("charge", item -> item * 2)   // Integer here
       .adapt(item -> "EUR " + item)          // Integer -> String
       .execute();                            // String (drop the adapt and it will not compile)
```

`execute()` without `just()` throws; `justAll` only exists on the root (injects through the shared chain, collect with `engine.await()`).

### WebFlux: infrastructure.reactive

`ReactiveFlow extends NioFlow`, `ReactiveStep extends NioStep`, `ReactiveLane extends Lane` тАФ a **subinterface mirror**, not a wrapper API: `Reactive.flow(DefaultNioFlow.from(Order.class))` and then `handleMono` / `adaptMono` / `adaptFlux` / `fanOutMono` / `executeMono` chain like any other step, with `pipe` / `pipeOrdered` / `pipeResilient` on the flow for a `Flux`. See `docs/rfc/0002-webflux.md`, `docs/rfc/0003-reactive-hardening.md` and `docs/rfc/0004-streaming-out.md`.

Load-bearing facts, all of them verified:

- **`defaultBudget(Duration)` is what stands between a hung socket and a leaked thread.** `Blocking.await` is `mono.block()`: with no budget the worker parks FOREVER on a Mono that never completes, and the engine has no cancellation to take it back. The default is declared once on the flow and applies to every reactive step that declared none тАФ on the main line, in a `just()` pipeline, inside a branch and inside a fork (`Lanes.budgeted` wraps the lane lambda and the segment body before core builds the lane; core ignores what a lane lambda returns, so this is invisible to it, and `Reactive.lane` then finds a budgeted lane already). It is `mono.timeout(d)`: the subscription is CANCELLED and `recover()` sees a `TimeoutException`, exactly as with an explicit per-step budget (which always wins). Opt-in on purpose тАФ a `Mono.just` needs no budget тАФ but a flow that talks to the network and declares none is one hung connection away from a worker lost for the life of the JVM.
- **`pipe` fails the whole stream on one bad element; `pipeResilient` does not.** Right for a request/response `Flux`, almost never right for an ingestion loop (Kafka, SSE, batch import) where one poison message would stop the consumer. `pipeResilient(concurrency, pipeline, onElementError)` reports the element and drops it; the handler is a mandatory parameter, not an overload тАФ dropping an element must be a decision, not a discovery. The engine's own `onError` handlers still see the failure, once. `concurrency`/`prefetch` are validated at BUILD time (Reactor would only reject them at the first element, inside `FluxFlatMap`).
- **A value is one object: streaming OUT is `executeFlux`, not a bigger `List`.** `adaptFlux(call)` has no cap тАФ ten million rows are an OOM inside a stage. `adaptFlux(call, maxItems)` bounds it: `take(maxItems + 1)` **cancels** the source, so an overrun costs one element, and the `FlowOverflowException` is an ordinary stage failure `recover()` catches. `executeFlux(tail)` is `executeMono().flatMapMany(tail)` тАФ the engine's one value, then Reactor's many, nothing buffered; it inherits laziness, one-execution-per-subscription, the empty-Flux filter cut and onError-before-the-tail-subscribes. The un-capped overload stays (removing it breaks every call site), but the rule is: if you cannot name a bound, do not collect it.
- **BlockHound guards the example's whole test suite** (test scope only, never core). The virtual workers need no allowlist тАФ they are not `NonBlocking` threads, and allowing blocking inside `Blocking.await` would mask the one bug worth catching. Instead the **bosses** are marked non-blocking, so "the boss never blocks" is mechanical: `BlockHoundTest` plants a `Thread.sleep` in a boss-inlined `handleSync` and asserts it trips (the engine catches `Throwable`, so the `Error` reaches `recover()` rather than killing the boss task). Two allowances, neither of them nioflow's: the boss's idle park on its task queue, and Jackson's `DeserializerCache` lock on Netty's decode thread.

- **The context bridge is declared, never discovered.** `propagate(TRACE, TENANT)` on the flow lifts those keys тАФ and *only* those тАФ out of Reactor's subscriber context into the per-execution `Context`, on every `executeMono`/`executeFlux`. Keys line up by NAME (`Context.Key` is a `record Key<T>(String name)`, which is why there is no cast anywhere). No `Hooks`, no Micrometer context propagation, no write-back: a reader of `NioFlowConfig` sees exactly what crosses. Seeded inside a `deferContextual`, so it is **per subscription** тАФ and that is why it does NOT go through `NioStep.with()`, which is a builder step that writes into the *pipeline* (two subscriptions would race on one map, and a key one carries would linger for the other). It runs through the core terminal `NioStep.executeAsync(Map<String, Object>)` instead: the run's entries merge with the pipeline's seed into a fresh map, `with()` wins on a shared name, and null/empty allocates nothing. Declaring no keys costs exactly zero (one branch on an empty list тАФ `monoOverhead` is unchanged). See `docs/rfc/0005-reactor-context-bridge.md`.
- **A reactive stage is not a new link.** `handleMono` appends the same `Stage` every other step appends, whose function parks a virtual worker on the Mono (`Blocking.await`). So it fuses, retries, rate-limits, lands in a lane and reports its metrics *because it is an ordinary stage*. The engine never learns what a `Mono` is; `core.model` stays sealed over the same eight links. Measured: four `handleMono` stages run at 57.1 ops/ms vs 56.0 for four plain `handle`s тАФ they fuse into one worker run, 2 hops for the four calls.
- **`Blocking.await` unwraps Reactor's `ReactiveException`.** `Mono.block()` wraps everything it throws, so without this a `mono.timeout(d)` would reach `recover()` as a Reactor wrapper instead of a `TimeoutException` тАФ and a reactive stage would stop looking like an ordinary stage, which is the one thing the facade promises.
- **`handleMono(name, call, budget)` тЙа `handle(name, fn, timeout)`.** The budget goes on the Mono (`mono.timeout`), which *cancels* the subscription so reactor-netty releases the connection. A stage timeout only abandons the parked worker; the HTTP request stays alive on the pool.
- **The methods are named, not overloaded.** `handle(String, UnaryOperator<T>)` + `handle(String, Function<T, Mono<T>>)` makes every existing implicit lambda ambiguous тАФ a source-breaking change (same trap as `handleContextual`). Hence `handleMono`, in the `handleSync`/`handleContextual` naming family.
- **The one wall: lanes.** `Condition.then` hard-codes `UnaryOperator<Lane<I>>`, and a reactive variant has the same erasure тАФ a name clash, not an override тАФ so a `ReactiveCondition` cannot both BE a `Condition` and hand out a `ReactiveLane`. Inside a `when`/`match` lambda you unwrap once with `Reactive.lane(lane)`. That is the only static helper that survived.
- **Two tests guard the design**: `ReactiveMirrorTest` reflects over `NioFlow`/`NioStep`/`Lane` and fails the build if a core method has no covariant override on its mirror (otherwise a chain silently falls back to the base type and loses its reactive steps) **and if a reactive-only step of `ReactiveStep` is missing from `ReactiveLane`** тАФ the covariance check says nothing about the reactive-only methods, and they drifted apart exactly once (`adaptMono(call, budget)` existed on the step and not on the lane, so a remote call inside a branch or a fork could not take a budget); `CoreWithoutReactorTest` runs a flow under a classloader that hides `reactor.**`, so core's zero-required-dependency promise cannot rot into a user's `NoClassDefFoundError`.
- **The trade-off, measured** (`ReactiveHeapProbeTest`): an in-flight request parked on a remote call retains **3 615 B** (an `Execution` + a parked virtual thread's stack) against **215 B** for a pure Reactor chain тАФ 16.8├Ч. Fine at 1 000 concurrent; at 100 000 it is 360 MB vs 21 MB. If every stage is a remote call AND concurrency is very high AND none of the engine (retry, rate limit, batch, key, fork, recover, splice, metrics, validation) is used тАФ recommend plain Reactor. The RFC has the decision tree.

`NioStep.with(Context.Key, value)` (core, no Reactor in the signature) seeds the per-execution context before the pipeline runs тАФ tracing needs it, and `just()` previously always passed a null context.

### Detached sub-flows: fork()

`fork(name, segment)` hands the value to a **child execution and keeps walking** тАФ the main line never waits for it (not even for its first boss-inlined link), its latency is not the request's, and a failure it does not `recover()` reaches `onError`, never the caller's future. Need the result back? That is `fanOut`; `fork` is the fire-and-forget *pipeline* the model was missing next to `background` (the fire-and-forget *step*). Available on `NioFlow`, `NioStep` and `Lane`, all type-preserving on the main line (the sub-flow's type never comes back).

The sub-flow builder is `Lane<T>` and its body is a `Segment` тАФ so a fork is a full pipeline (`handle`/`handleSync`/`handleContextual`/`background`/`adapt`/`filter`/`recover`/`fanOut`/`batch`/`use`/`when`/`match`/nested `fork`), reusable and testable on its own, with no new builder interface. Design notes that are load-bearing (see `docs/rfc/0001-fork.md`):

- The parent submits the child as a **new boss task** (`boss.execute(child)`), never inline тАФ inlining would make the main line wait for the fork's boss-side links.
- The child gets a **copy** of the context: parent and child run concurrently, and the plain `HashMap` behind `Context` is only safe because ONE execution's stages are serialized by its executor handoffs. Writes inside a fork stay inside the fork.
- The child is **never keyed** (inheriting the key would queue it behind its own parent) and **never notifies `completeHandlers`** (its terminal value is not the flow's `O`); it reports through `forkStarted`/`forkCompleted`/`forkFailed`/`forksInFlight` instead of the execution metrics.
- The `Fork` LINK carries the caller's guards (a fork inside a lane only spawns for values routed there); the links INSIDE start from no guards, so the sub-chain is **guard-closed** and its decision ids are compacted to `0..n-1` (they are private, and the engine-wide counter would bloat the child's bitset).
- Forks are **in-flight work**: `spawnFork` bumps the drain counter, so `shutdown(grace)` waits for them. The drain slot is released only after the execution's metrics and handlers ran ("in flight" = "not fully reported"), which is what makes a clean drain mean "everything reported".
- `Fork` is a dispatch boundary (it breaks fused runs, like `Background`); its name is a splice anchor, so a runtime `splice(name, REPLACE, ...)` swaps the whole sub-flow atomically. Measured: main-line throughput and allocation unchanged when no fork is declared; with one, the cost is a `background`'s plus one child `Execution`.

### Branching: when() and match()

Both are sugar over `Decision`/`Guard` тАФ the engine knows nothing about branching. A lane is a *guarded view* of the same chain (`AbstractChain.withGuards`) wrapped in the restricted `Lane<T>` interface: every link declared inside the lane's `UnaryOperator<Lane<T>>` carries `Guard(decisionId, expected)`, so nested branches compose guards automatically and branching works identically on the shared definition and inside a `just()` execution.

- `when(pred).then(lane).otherwise(lane)`: appends one `Decision`; the lanes require `true`/`false`. Chaining after the fork (on `Branch` or after `otherwise`) is the main line тАФ unguarded, runs for every value.
- `match().is(pred, lane)...otherwise(lane)` is **first-match-wins**: case k's `Decision` carries guards requiring all previous cases `false` (so it isn't even evaluated after a match), its lane additionally requires its own decision `true`, and `otherwise` requires all `false`. A skipped `Decision` records nothing, and an absent decision fails any guard on it тАФ that's what makes the semantics hold.
- `Lane<T>` exposes only step-building operations (`handle`/`background`/`adapt`/`filter`/`recover`/`fanOut`/`use` and nested `when`/`match` via `LaneCondition`/`LaneBranch`/`LaneCases`) тАФ no `just`, no `execute`, no `close`. The type going into and out of a lane is `T` (an `adapt` inside a lane must return to `T`, enforced by the `UnaryOperator` signature).
- **Sub-flows**: `Segment<T, R>` defines a reusable chain piece over `Lane<T>` ending at `Lane<R>`; `use(segment)` embeds it inline with the caller's guards (lane-scoped inside branches), segments compose via `lane.use(...)`, and they are build-time only тАФ zero runtime footprint (the embedded links compile/fuse like inline ones). Prefer segments to keep large shared definitions readable (see the example's `NioFlowConfig`: platform / fraudGate / shippingTier / notifications).
